Sensi v2: Teaching an Agent to Learn One Thing at a Time

Programmable context, curriculum learning, and what happens when your agent learns everything wrong — in 32 tries

Mohsen Arjmandi · Freddie Spirit · March 2026

Previously on this blog

In Solving ARC-AGI-3 with Senses, I introduced Sensi v1 — an LLM agent that plays the ARC-AGI-3 games by splitting cognition into two players: an Observer (Player 1) who watches the screen and maintains lists of guesses and figured-out items, and an Actor (Player 2) who picks the next game action based on those lists. The idea was simple: if an LLM can't see and act at the same time, give it two roles that communicate through structured lists, like two friends playing a co-op game over the phone.

Sensi v1 managed to solve 2 levels of the ARC-AGI-3 challenge. It was a proof of concept that LLM agents can learn game mechanics through trial and error, rather than being pre-programmed with game knowledge. But v1 had a fundamental problem: the agent tried to learn everything at once. It would guess about actions, game rules, and win conditions all in a single muddled stream of consciousness. There was no curriculum, no focus, no way to tell the agent "first learn this, then learn that."

This post is about Sensi v2 — a redesign that introduces curriculum-based learning, programmable context via a database, and a sense scoring system that lets an external judge decide when an agent has learned enough about one topic to move to the next. The results were surprising: v2 didn't win any games, but it revealed two things that I believe matter more than any leaderboard score.

Background: The ARC-AGI-3 Problem

ARC-AGI-3 is a game-based evaluation where agents must play pixel-art puzzle games without prior knowledge of the rules. The agent receives a grid of colored pixels (the game frame) and can execute actions (move up, down, left, right, interact, click, etc.). It must figure out what the game is, what each action does, and how to win — all from scratch.

The game state is represented as a 3D tensor:

$$\mathcal{F}_t \in \mathbb{Z}_{16}^{L \times H \times W}$$

where $L$ is the number of grid layers, $H$ and $W$ are the grid dimensions (up to 64×64), and each cell value is an integer in $[0, 15]$ representing a color. The agent has access to a discrete action space:

$$\mathcal{A} = \{a_0, a_1, \ldots, a_7\}$$

where $a_0$ is RESET, $a_1$ through $a_5$ are simple actions (directional moves, interact), $a_6$ is a complex action taking $(x, y)$ coordinates, and $a_7$ is an additional simple action. At each timestep the agent observes $\mathcal{F}_t$, selects $a \in \mathcal{A}$, and receives $\mathcal{F}_{t+1}$ along with a scalar score $s_t \in [0, 254]$.

What Sensi v1 Did

Sensi v1 split the agent into two DSPy modules communicating through shared lists. Each turn looked like this:

$$\text{Player}_1(\mathcal{F}_t, \mathcal{F}_{t-1}, a_{t-1}, \Delta_t) \;\rightarrow\; (G_t, K_t)$$ $$\text{Player}_2(G_t, K_t) \;\rightarrow\; (d_t, a_t)$$

where $G_t$ is the guesses list, $K_t$ is the figured-out list, $d_t \in \{\text{GUESS}, \text{INFORMED}\}$ is the decision type, and $\Delta_t$ is a visual diff between consecutive frames produced by a separate LLM call. The state was persisted across game resets using an SQLite database, so the agent could learn from previous failed attempts.

The key insight from v1 was that this two-player architecture actually works: the Observer generates hypotheses, the Actor tests them, and over multiple plays the figured-out list converges toward correct game knowledge. But the convergence was slow and unfocused, because the agent was trying to learn actions, rules, and win conditions all simultaneously.

The Design of Sensi v2: Curriculum Learning for Game Agents

The core idea behind v2 is embarrassingly simple: learn one thing at a time.

Instead of letting the agent wander through an unstructured exploration space, v2 introduces a queue of learning items — topics the agent must master sequentially. Each item has a state machine, a dynamically generated evaluation metric, and a sense score that determines when the agent can graduate to the next topic.

The Learning Queue

At the start of each game, Sensi v2 initializes a learning curriculum stored in an SQLite table:

def initialize_items_to_learn(self, game_id: str, card_id: str) -> None:
    # Items that are already known facts (don't need to learn)
    fact_items = [
        "RESET starts the game",
        "all available actions: ACTION1, ACTION2, ..., ACTION7, RESET",
    ]

    # Items that need to be learned, in order
    default_items = [
        "learn what each action does in the game",
        "learn how actions affects your energy while playing",
        "learn how to win the game",
    ]

    for item_name in default_items:
        cur.execute("""
            INSERT OR IGNORE INTO items_to_learn
            (game_id, card_id, item_name, state, threshold)
            VALUES (?, ?, ?, 'not_reached', 8)
        """, (game_id, card_id, item_name))

The agent starts with some facts (known truths like "RESET starts the game") and a sequence of learning items it must tackle one by one. The design is intentionally modular: you can change the curriculum by editing this list, and the agent's behavior changes accordingly without touching any prompt logic.

The State Machine

Each learning item follows a three-state lifecycle:

    ┌─────────────────┐
    │   not_reached    │  (waiting in queue)
    └────────┬────────┘
             │  picked as next item
             ▼
    ┌─────────────────┐
    │    learning      │◄────── score < threshold
    └────────┬────────┘         (keep trying)
             │  score ≥ threshold
             ▼
    ┌─────────────────┐
    │   completed      │  → figured_out items become new FACTS
    └─────────────────┘

Formally, let $\mathcal{I} = \{I_1, I_2, \ldots, I_n\}$ be the ordered sequence of learning items, each with state $\sigma_i \in \{\texttt{not\_reached}, \texttt{learning}, \texttt{completed}, \texttt{fact}\}$. At any turn $t$, the active item is:

$$I^* = \min_{i} \{I_i \mid \sigma_i \in \{\texttt{learning}, \texttt{not\_reached}\}\}$$

When $I^*$ transitions to completed, its accumulated figured_out entries are promoted to facts:

$$\sigma_i = \texttt{completed} \;\Longrightarrow\; \forall\, k \in K_t : \sigma_k \leftarrow \texttt{fact}$$

These new facts then become part of the context for subsequent learning items, creating a knowledge accumulation chain.

The Sense Scorer: LLM-as-Judge for Learning Progress

How does the agent know when it has learned enough about a topic? V2 introduces a sense scorer — a separate LLM call that acts as an external judge. But the judge doesn't use a fixed rubric. Instead, the rubric itself is generated dynamically by another LLM call.

The two-phase evaluation works like this:

Phase 1: Metric Generation. When a new learning item becomes active, a MetricGeneratorSignature is invoked to produce a verification criterion:

class MetricGeneratorSignature(dspy.Signature):
    """Given an item the agent needs to learn about a game,
    generate a metric to verify that the item has been learned."""

    item_to_learn: str = dspy.InputField(
        desc="The item/concept the agent needs to learn"
    )
    learning_metric: str = dspy.OutputField(
        desc="A criteria and description that a judge will use to give a
         score on how good a grasp the learner has on the item.
         Score between 1 to 10."
    )

Phase 2: Sense Scoring. On each subsequent turn, the SenseScorerSignature evaluates the agent's current understanding against that metric:

class SenseScorerSignature(dspy.Signature):
    """Score the agent's understanding of a learning item from 1 to 10."""

    item_to_learn: str = dspy.InputField()
    learning_metric: str = dspy.InputField()
    facts: List[str] = dspy.InputField()
    figured_out: List[str] = dspy.InputField()

    sense_score: int = dspy.OutputField(
        desc="Score from 1-10 indicating learning progress."
    )
    reasoning: str = dspy.OutputField(
        desc="Brief explanation for the score"
    )

This creates a feedback loop. Let $\mu_i$ denote the metric generated for item $I_i$, and $\phi_t$ the sense score at turn $t$. The transition condition is:

$$\phi_t(I^*, \mu_{I^*}, \mathcal{K}_t, K_t) \;\geq\; \tau_{I^*} \;\;\Longrightarrow\;\; \sigma_{I^*} \leftarrow \texttt{completed}$$

where $\mathcal{K}_t$ is the set of accumulated facts and $\tau_{I^*}$ is the threshold (default 8 out of 10). The sense reasoning $r_t$ is fed back to Player 1 on the next turn, telling it why the score was what it was and what's still missing.

Programmable Context: The Database as a Control Plane

One of the most architecturally significant choices in v2 is using an SQLite database as the agent's programmable context. Every turn, the agent reads its state from and writes its state to several database tables:

TablePurpose
items_to_learnLearning curriculum with state machine
inputsKey-value store for per-turn game state
gameTurn-by-turn history of frames, actions, diffs
guessesPlayer 1's hypothesis lists per turn
figured_outsPlayer 1's confirmed observations per turn
losing_actions_seqsAction sequences that led to game over

This is not just persistence — it's a control plane. By modifying what's in the database, you change the agent's behavior without changing any code or prompts. Want the agent to skip action learning and go straight to win-condition exploration? Delete those rows. Want to seed it with facts from a human player? Insert them. Want to change the learning order? Reorder the items.

The implication is profound: the context window of the LLM becomes programmable through database state. We can write to the database, and those writes become part of the agent's next prompt through the get_facts(), get_current_item_to_learn(), and get_action_history() calls. Formally:

$$\text{Prompt}_t = f\!\left(\mathcal{F}_t,\; \text{DB}.\texttt{facts},\; \text{DB}.\texttt{item\_to\_learn},\; \text{DB}.\texttt{history},\; \text{DB}.\texttt{sense\_score}\right)$$

This makes the entire learning process externally steerable. The LLM doesn't need to hold a curriculum in its context window; the curriculum lives in the database and is injected turn by turn. This is a fundamentally different architecture from putting everything in a system prompt.

The Full Turn Architecture

Putting it all together, here is the complete data flow for a single turn in Sensi v2. Each turn involves up to five LLM calls organized in a pipeline:

$$\underbrace{\text{FrameDiff}}_{\text{LLM}_1} \;\rightarrow\; \underbrace{\text{MetricGen}}_{\text{LLM}_2} \;\rightarrow\; \underbrace{\text{SenseScore}}_{\text{LLM}_3} \;\rightarrow\; \underbrace{\text{Player}_1}_{\text{LLM}_4} \;\rightarrow\; \underbrace{\text{Player}_2}_{\text{LLM}_5}$$

Step by step:

1. Frame Differencing ($\text{LLM}_1$). The raw grid $\mathcal{F}_t$ is rendered to a scaled pixel image via a color map ($\mathbb{Z}_{16} \to \text{RGB}$). Two consecutive frames are passed to a multimodal LLM that returns a structured JSON diff:

def frame_diff_finder(self, current_frame, prev_frame) -> str:
    prediction = self.frame_diff_module(
        prev_frame=DSPyImage(url=encode_image(prev_frame)),
        current_frame=DSPyImage(url=encode_image(current_frame)),
    )
    return prediction.diff_json  # structured JSON

The diff schema captures added/removed/moved objects, UI changes, and a high-level summary — giving the observer a structured description of what changed rather than forcing it to compare raw pixels.

2. Metric Generation ($\text{LLM}_2$). Only called once per learning item, when a new item enters the learning state. Generates a rubric for evaluation.

3. Sense Scoring ($\text{LLM}_3$). Called every turn for the active learning item. Produces a score $\phi_t \in [1, 10]$ and reasoning text.

4. Player 1 — Observer ($\text{LLM}_4$). Receives the game frame, frame diff, previous guesses/figured-out, facts, current learning target, sense score feedback, and action history. Outputs updated guesses and figured-out lists. The prompt includes detailed guidelines on list curation: keep valid items, edit with new info, remove contradicted ones, promote confirmed guesses.

5. Player 2 — Actor ($\text{LLM}_5$). Receives guesses, figured-out, facts, and the current learning target. Outputs a decision type (GUESS/INFORMED) and exactly one action. The actor's job is to resolve the observer's doubts by choosing actions that help move guesses to the figured-out list.

Modeling the Learning Process

We can frame the entire Sensi v2 learning process as a partially observable Markov decision process (POMDP) augmented with an external knowledge base.

Define the agent's epistemic state at time $t$ as the tuple:

$$\mathcal{E}_t = \left(\mathcal{K}_t,\; G_t,\; K_t,\; I^*_t,\; \phi_t,\; \mu_{I^*_t}\right)$$

where $\mathcal{K}_t$ is the fact set, $G_t$ is guesses, $K_t$ is figured-out, $I^*_t$ is the active learning item, $\phi_t$ is the sense score, and $\mu_{I^*_t}$ is the evaluation metric. The transition dynamics are:

$$\mathcal{E}_{t+1} = T\!\left(\mathcal{E}_t,\; a_t,\; \mathcal{F}_{t+1},\; \text{LLM}_1(\mathcal{F}_t, \mathcal{F}_{t+1}),\; \text{LLM}_3(\cdot),\; \text{LLM}_4(\cdot)\right)$$

The policy $\pi$ is the composition of all five LLM modules:

$$a_t = \pi(\mathcal{E}_t, \mathcal{F}_t) = \text{LLM}_5\!\Big(\text{LLM}_4\!\big(\mathcal{F}_t, \Delta_t, \mathcal{E}_t\big)\Big)$$

What's unique about this formulation is that the reward signal is internal. The agent doesn't optimize for the game score directly. Instead, it optimizes for the sense score $\phi_t$ on the current learning item. The game score is an emergent property of learning the right things in the right order. This is analogous to intrinsic motivation in reinforcement learning, but implemented through LLM-based self-evaluation rather than curiosity-driven exploration bonuses.

Results: Everything Wrong, Everything Right

Sensi v2 was run on ARC-AGI-3 games using Gemini 3.1 Pro as the backbone model. The results were, on the surface, disappointing:

Score: 0 levels won (vs. 2 levels in v1)

The agent learned all listed topics — game actions, game rules, win conditions — properly navigating the curriculum. But it learned the wrong things throughout. The figured-out items were confidently stated but factually incorrect descriptions of what each action does, how the game rules work, and how to win.

However, two findings emerged that I believe are genuinely significant:

Finding 1: Programmable Context Works

The agent was properly guided through the entire curriculum by commands stored in the database. The state machine transitioned correctly: it started with action learning, the sense scorer evaluated progress, when the threshold was reached the figured-out items were promoted to facts, and the agent moved to the next learning item. This happened automatically across multiple game resets.

This proves that you can control an LLM agent's behavior through database state rather than through prompt engineering alone. The curriculum, the evaluation metrics, and the learning progression are all external to the model — they're data, not code. This is a form of neuro-symbolic programming: the symbolic structure (the curriculum, the state machine, the database) guides the neural system (the LLM) through a structured learning process.

Finding 2: 32 Tries — Human-Level Sample Efficiency

The agent completed its entire learning curriculum in approximately 32 action attempts. It learned (incorrectly, but systematically) what each action does, how the energy system works, and the win condition — all within 32 interactions with the game environment.

For comparison, consider Agentica from Symbolica, which reportedly requires 1,600 to 3,000 tries per game to build up an understanding. Sensi v2 achieved curriculum completion in roughly $\frac{1}{50}$th to $\frac{1}{100}$th of that budget.

$$\text{Sample efficiency ratio} = \frac{N_{\text{other}}}{N_{\text{Sensi}}} = \frac{1600\text{--}3000}{32} \approx 50\text{--}94\times$$

Even though Sensi v2 learned the wrong things, the speed of learning is what matters for the broader research agenda. If an agent architecture can learn from such a small number of examples, then the bottleneck is not sample efficiency but grounding — ensuring that what the LLM perceives and concludes is actually correct. This is a much more tractable problem than the sample efficiency problem.

In the context of test-time inference research, this is especially relevant. Test-time compute is expensive. An architecture that needs 32 interactions to form a hypothesis about a new task is fundamentally different from one that needs 3,000. If we can fix the grounding problem (better vision, better frame differencing, better self-verification), the same 32-try architecture could produce correct answers.

Why Did It Learn Wrong Things?

The failure mode is instructive. The frame differencing step ($\text{LLM}_1$) produces a natural language description of visual changes between frames. Gemini 3.1 Pro, when looking at small pixel-art frames scaled to 10x, often misidentifies objects, positions, and movements. A player moving left might be described as "object shifted right." A UI element showing remaining energy might be described as "decorative border pattern."

These errors propagate through the entire pipeline. Player 1 builds guesses based on incorrect frame diffs. The sense scorer evaluates understanding based on what Player 1 has figured out — but since the figured-out items are internally consistent (just wrong), the scorer gives high marks. The agent confidently "learns" a hallucinated game model and moves on to the next curriculum item.

This is a self-consistent hallucination cascade:

$$\Delta_t^{\text{wrong}} \;\rightarrow\; K_t^{\text{wrong}} \;\rightarrow\; \phi_t^{\text{high}} \;\rightarrow\; \sigma_{I^*} = \texttt{completed} \;\rightarrow\; \mathcal{K}_{t+1}^{\text{wrong}}$$

Each learning item builds on the (wrong) facts from the previous item, compounding the error. The architecture is correct — the information flow is exactly right — but the perception layer is the weak link.

What This Means for Neuro-Symbolic AI

Sensi v2 sits at the intersection of several active research areas:

Curriculum learning for LLM agents. Most LLM agents tackle tasks monolithically. Sensi v2 shows that breaking a task into a sequence of learnable sub-goals and managing them through an external state machine can produce structured, efficient learning — even with imperfect perception.

LLM-as-judge for learning verification. The sense scoring system demonstrates that LLMs can serve as internal evaluators of their own learning progress. The two-phase approach (generate metric, then score against it) is more flexible than fixed rubrics and can adapt to arbitrary learning domains.

Programmable agent context. The database-as-control-plane pattern is, I believe, underexplored. Most agent frameworks manage context through prompt templating. Sensi v2 shows that externalizing the cognitive state to a database and programmatically injecting it into prompts creates a more modular, steerable, and debuggable agent architecture.

Sample efficiency in test-time inference. The 32-try result suggests that the right architecture can dramatically reduce the number of interactions needed to form hypotheses about new environments. This has implications beyond games: any test-time inference scenario (novel tool use, unfamiliar APIs, new document formats) could benefit from architectures that learn from minimal examples.

What's Next

The path forward is clear: fix the perception layer. The curriculum system, the state machine, the sense scoring, the two-player architecture — all of these work. The bottleneck is the quality of $\Delta_t$, the frame diff. Better visual grounding — whether through fine-tuned vision models, structured pixel analysis, or hybrid approaches that combine LLM perception with programmatic pixel comparison — should allow the same architecture to learn correct game models within the same 32-try budget.

The question Sensi v2 raises is not "can LLM agents learn games?" but "given that they can learn efficiently, how do we make sure they learn correctly?" That's a much better question to be asking.
